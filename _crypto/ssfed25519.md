# ssfed25519 — Ed25519 Signatures

[SSF](../README.md) | [Cryptography](README.md)

Ed25519 digital signatures over the edwards25519 twisted-Edwards curve, per RFC 8032
("pure" EdDSA variant only). Application-facing signature scheme paired with the X25519 key
agreement exposed by [`ssfx25519`](ssfx25519.md); the two share a base curve
(Curve25519 / edwards25519) but use different coordinate systems and are separate wire
formats.

Compared with [ECDSA over NIST curves](ssfecdsa.md), Ed25519 has a few distinct
engineering properties:

- **Always deterministic.** RFC 8032 signing is deterministic by construction, not by
  convention: the per-signature nonce is `SHA-512(prefix ‖ message)` where `prefix` is
  derived from the long-term seed. No RNG at sign time, and no mode / parameter choice to
  get wrong.
- **Fixed, compact wire format.** 32-byte seed, 32-byte public key, 64-byte signature. No
  DER wrapping, no ASN.1 OIDs, no length fields.
- **Signing hashes the raw message internally (SHA-512).** Unlike ECDSA — where the caller
  hashes first — Ed25519 accepts the message bytes directly and hashes them as part of the
  construction. No hash-alignment choice to make.
- **Self-contained field arithmetic.** Like [`ssfx25519`](ssfx25519.md), this module
  implements GF(2²⁵⁵ − 19) directly and does not depend on [`ssfbn`](ssfbn.md). The only
  cross-module dependency is [`ssfsha2`](ssfsha2.md) for SHA-512.

This module implements the `Ed25519` "pure" signing variant only. The `Ed25519ctx` and
`Ed25519ph` variants from RFC 8032 §5.1.6 (which add a domain-separation context and
optional pre-hashing) are not provided — if your protocol requires one of those, it must
be out of reach for now.

  nonce prefix, to derive the deterministic per-signature nonce, and to compute the
  challenge hash `SHA-512(R ‖ A ‖ M)`.

<a id="notes"></a>

## [↑](#ssfed25519--ed25519-signatures) Notes

- **The 32-byte seed is the secret — treat it as the private key.** Ed25519 expands the
  seed once per sign or public-key-derive call: `h = SHA-512(seed)`, `a =
  clamp(h[0..31])`, `prefix = h[32..63]`. The scalar `a` is what actually signs; the
  prefix is what makes nonce derivation deterministic. Callers store and protect the
  32-byte seed, not the expanded key. Zeroize the seed as soon as its last consumer is
  done.
- **`Sign` takes both seed *and* public key; there is no internal consistency check.**
  Ed25519 implementations commonly store `(seed, pubKey)` as a single "secret key" blob
  (64 bytes total, per RFC 8032 §5.1.5's reference encoding) so that sign can avoid
  re-deriving the public key on every call. This module follows that pattern. If the
  caller passes a `pubKey` that does not match the `seed`, the resulting signature is
  simply wrong — `SSFEd25519Sign` does not detect or report the mismatch. Produce the
  public key once via [`SSFEd25519KeyGen`](#ssfed25519keygen) or
  [`SSFEd25519PubKeyFromSeed`](#ssfed25519pubkeyfromseed) and always pass it alongside
  its seed thereafter.
- **Signing is deterministic.** The RFC-8032 nonce derivation is
  `SHA-512(prefix ‖ msg) mod L`. Signing the same `(seed, pubKey, msg)` twice yields
  identical signatures. This eliminates the nonce-reuse disaster class that plagues
  stateful / RNG-based ECDSA implementations, and means there is no entropy requirement
  at sign time — once the seed is in hand, signing is pure function.
- **Key-generation entropy is platform-dependent.** [`SSFEd25519KeyGen()`](#ssfed25519keygen)
  reads 32 bytes directly from `/dev/urandom` on POSIX; on Windows the internal
  `_SSFEd25519GetEntropy()` trips `SSF_ASSERT(false)` and returns `false`. On embedded
  targets, patch that helper to a TRNG or provision seeds off-device and use
  [`SSFEd25519PubKeyFromSeed()`](#ssfed25519pubkeyfromseed) at runtime. **Never** ship
  this module's `KeyGen` on a target without a genuine entropy source — seed
  predictability breaks every signature the device ever produces.
- **Wire formats are simple.** Seed and public key are each 32 bytes, signature is 64
  bytes — `R ‖ S` with both halves 32 bytes and `R` encoded as a compressed Edwards-curve
  point (y-coordinate with sign bit) per RFC 8032 §5.1.6. No DER encoding, no explicit
  length anywhere. Serialize at the call site as raw bytes.
- **Sign signs the raw message.** Pass the message bytes to [`SSFEd25519Sign()`](#ssfed25519sign)
  directly; do *not* pre-hash. The SHA-512 hashing happens inside the construction — if
  you pass a SHA-256 or SHA-512 digest as the message, you will produce a signature over
  the digest, not over the original data, and a verifier given the original data will
  fail. For the (rare) case where you need to sign a hash because the message cannot fit
  in memory, use the `Ed25519ph` variant — which this module does not implement.
- **Signing is constant-time; verification is intentionally not.** The sign path routes
  through `_ge_scalarmult_ct` (ssfed25519.c:685), a double-and-add-always scalar
  multiplication: each of the 256 scalar-bit iterations does exactly one `_ge_double`,
  one `_ge_add` into a scratch point, and one constant-time `_ge_cswap` (built on the
  module's mask-XOR `_fe_cswap`). Wall-clock runtime per scalar multiplication is
  independent of the secret nonce bits, closing the Hamming-weight timing channel that
  the biased-nonce lattice-attack class (Minerva, TPM-FAIL) has used to recover ECC
  private keys from timing-biased sign operations on real hardware. Sign is therefore
  roughly 2× slower than a naive variable-time double-and-add; a precomputed base-point
  table (the libsodium / BoringSSL approach) could restore the speed without giving up
  the constant-time property, but this module does not currently include one. Verify
  continues to use the variable-time `_ge_scalarmult` (ssfed25519.c:651) via
  `_ge_double_scalarmult` — both of its scalars are derived from the public signature
  and message, so the timing channel there reveals no secret. The final `R == R'`
  comparison in verify uses [`SSFCryptCTMemEq`](ssfcrypt.md) as defense-in-depth against remote
  timing oracles. Signing zeroes the seed-derived intermediates (`h`, `a`, `nonce`,
  `r_hash`, the SHA-512 context) on return to limit exposure of stale state.
- **No explicit scalar clamping at the API.** Clamping of `a` happens inside the seed
  expansion; callers never see it. Unlike [`ssfx25519`](ssfx25519.md), different seeds
  *do* produce different public keys — the ambiguity that two clamped-equivalent X25519
  private keys collapse into one public key does not apply here.
- **Verification rejects any signature with `S ≥ L`.** Per RFC 8032 §5.1.7, verifiers must
  reject the signature if `S` is not in `[0, L-1]` where `L` is the curve order. This
  module enforces that check before doing any point arithmetic. Malleability is bounded:
  a valid `(R, S)` cannot be trivially perturbed into another valid signature on the same
  message.
- **Ed25519 is specified as Ed25519 only.** There is no curve parameter to pass — one
  curve, one hash, one everything. The 32 / 32 / 64 byte sizes and the SHA-512 choice are
  wired into the scheme.

## [↑](#ssfed25519--ed25519-signatures) Function Reference

<a id="ssfed25519keygen"></a>

### [↑](#functions) [`bool SSFEd25519KeyGen()`](#functions)

```c
bool SSFEd25519KeyGen(uint8_t seed[SSF_ED25519_SEED_SIZE],
                      uint8_t pubKey[SSF_ED25519_PUB_KEY_SIZE]);
```

Generate a fresh `(seed, pubKey)` pair. Reads 32 bytes from the platform entropy source
into `seed`, then calls [`SSFEd25519PubKeyFromSeed()`](#ssfed25519pubkeyfromseed) to
derive the matching public key. Returns `false` if entropy is unavailable.

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `seed` | out | `uint8_t[32]` | The 32-byte seed — treat as secret. |
| `pubKey` | out | `uint8_t[32]` | The derived public key. Compressed Edwards point. |

**Returns:** `true` on success; `false` if the platform entropy source is unavailable.

<a id="ex-keygen"></a>

**Example:**

```c
uint8_t seed[SSF_ED25519_SEED_SIZE];
uint8_t pub[SSF_ED25519_PUB_KEY_SIZE];

if (!SSFEd25519KeyGen(seed, pub))
{
    /* Platform entropy unavailable — on bare metal, wire
       _SSFEd25519GetEntropy() to a TRNG, or provision the seed at
       manufacture and use SSFEd25519PubKeyFromSeed at boot. */
}
/* Keep `seed` secret. Publish `pub`. Store them together so sign can
   be called without re-deriving pub every time. */
```

<a id="ssfed25519pubkeyfromseed"></a>

### [↑](#functions) [`void SSFEd25519PubKeyFromSeed()`](#functions)

```c
void SSFEd25519PubKeyFromSeed(const uint8_t seed[SSF_ED25519_SEED_SIZE],
                              uint8_t pubKey[SSF_ED25519_PUB_KEY_SIZE]);
```

Derive the public key from a 32-byte seed. Internally: hash the seed with SHA-512, clamp
the lower 32 bytes to form the scalar `a`, and encode `A = a·B` (the base-point scalar
multiplication) as the compressed Edwards point. Useful when the seed was provisioned from
elsewhere (secure element, factory programming) and the device needs its own public key
at boot.

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `seed` | in | `const uint8_t[32]` | The 32-byte secret seed. |
| `pubKey` | out | `uint8_t[32]` | The derived public key. |

**Returns:** Nothing. Cannot fail for any 32-byte input.

<a id="ex-derive"></a>

**Example:**

```c
extern const uint8_t factory_seed[32];
uint8_t pub[SSF_ED25519_PUB_KEY_SIZE];

SSFEd25519PubKeyFromSeed(factory_seed, pub);
/* pub is the device's Ed25519 public key — ready to embed in
   attestations, CSRs, or identity tokens. */
```

<a id="ssfed25519sign"></a>

### [↑](#functions) [`void SSFEd25519Sign()`](#functions)

```c
void SSFEd25519Sign(const uint8_t seed[SSF_ED25519_SEED_SIZE],
                    const uint8_t pubKey[SSF_ED25519_PUB_KEY_SIZE],
                    const uint8_t *msg, size_t msgLen,
                    uint8_t sig[SSF_ED25519_SIG_SIZE]);
```

Produce an Ed25519 signature over `msg`. Deterministic: identical inputs produce an
identical signature. Cannot fail for any well-formed input — the only way to misuse this
is to pass a `pubKey` that does not correspond to `seed`, which produces a signature that
fails verification.

The signing steps follow RFC 8032 §5.1.6 exactly: derive `(a, prefix)` from `SHA-512(seed)`,
compute the per-signature nonce `r = SHA-512(prefix ‖ msg) mod L`, encode `R = r·B` into
the first 32 bytes of the signature, compute the challenge `k = SHA-512(R ‖ A ‖ msg) mod L`,
and write `S = (r + k·a) mod L` into the last 32 bytes. Secret intermediates are zeroed on
return.

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `seed` | in | `const uint8_t[32]` | The 32-byte secret seed. |
| `pubKey` | in | `const uint8_t[32]` | The matching public key. Must correspond to `seed` — not checked. |
| `msg` | in | `const uint8_t *` | Message to sign (raw bytes — not a hash). May be `NULL` when `msgLen` is `0`. |
| `msgLen` | in | `size_t` | Number of message bytes. |
| `sig` | out | `uint8_t[64]` | Buffer receiving the 64-byte `R ‖ S` signature. |

**Returns:** Nothing.

<a id="ex-sign"></a>

**Example:**

```c
/* Sign a firmware-update manifest. Ed25519 signs the raw bytes, no
   caller-side hashing. */
const uint8_t *manifest;
size_t         manifestLen;

uint8_t sig[SSF_ED25519_SIG_SIZE];
SSFEd25519Sign(seed, pub, manifest, manifestLen, sig);

/* sig is 64 bytes, ready to ship alongside the manifest. Deterministic:
   signing the same manifest with the same seed produces the same sig. */
```

<a id="ssfed25519verify"></a>

### [↑](#functions) [`bool SSFEd25519Verify()`](#functions)

```c
bool SSFEd25519Verify(const uint8_t pubKey[SSF_ED25519_PUB_KEY_SIZE],
                      const uint8_t *msg, size_t msgLen,
                      const uint8_t sig[SSF_ED25519_SIG_SIZE]);
```

Verify an Ed25519 signature. Rejects signatures where `S ≥ L` (range check) or where the
public key fails to decode as a valid Edwards-curve point; then runs the verify equation
`[S]B − [k]A == R` using a variable-time double-scalar multiplication. The final
`R == R'` comparison is constant-time via [`SSFCryptCTMemEq()`](ssfcrypt.md).

**Returns:** `true` iff the signature is well-formed and verifies against `(pubKey, msg)`;
`false` otherwise. All `false` outcomes — range check failure, malformed public key,
equation mismatch — should be treated identically (reject the signed message).

<a id="ex-verify"></a>

**Example:**

```c
/* At the peer: verify the received firmware-update signature. */
if (!SSFEd25519Verify(trustedPub, manifest, manifestLen, sig))
{
    /* Signature invalid — manifest has been tampered with, the wrong
       key was used, or the sig is malformed. Abort the update. */
    return;
}
/* Accept and apply the update. */
```
