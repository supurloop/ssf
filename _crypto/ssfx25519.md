# ssfx25519 — X25519 Key Agreement

[SSF](../README.md) | [Cryptography](README.md)

X25519 elliptic-curve Diffie-Hellman key agreement over Curve25519, per RFC 7748 §5. A
self-contained implementation of the modern alternative to
[ECDH over NIST P-256 / P-384](ssfecdsa.md) — same use case (derive a shared secret from
two public keys), different curve, different engineering profile:

- **Fully constant-time.** The Montgomery ladder has no secret-dependent branches or memory
  accesses; all scalar-conditional steps use
  [constant-time swaps](#constant-time-notes). Contrast with NIST ECDSA / ECDH, where some
  paths (`SSFBNModMulNIST`) leak timing on public data and the Montgomery-ladder identity
  edge case has a documented caveat.
- **Smaller and faster on microcontrollers.** Self-contained GF(2²⁵⁵ − 19) field arithmetic
  (no dependency on [`ssfbn`](ssfbn.md)), ~500 LOC, ~5 KB flash, ~300 B peak stack.
- **One fixed key / secret size.** 32 bytes for everything — private key, public key, shared
  secret. No curve selection, no coordinate encoding variants.

Use X25519 when you get to choose the key agreement, and use
[`ssfecdsa`'s ECDH](ssfecdsa.md#ecdh) when you need interop with NIST-only peers or X.509
certificate chains that require P-256 / P-384 keys. X25519 is not paired with a signature
scheme in this module — for Ed25519 signatures see [`ssfed25519`](ssfed25519.h), which is
also a Curve25519-family primitive but uses a different curve representation (twisted
Edwards) and has its own keypair format.

## [↑](#ssfx25519--x25519-key-agreement) Notes

- <a id="constant-time-notes"></a>**Constant-time by construction.** The scalar
  multiplication uses a Montgomery ladder that processes every one of the 255 scalar bits,
  with a constant-time conditional swap of the two candidate projective points between
  iterations. The field arithmetic uses no table lookups, no data-dependent branches, and
  no variable-time shifts. Runtime is fixed per scalar-mul call, independent of the scalar
  or peer-point bytes. This is the main engineering reason X25519 was designed in the first
  place — unlike the NIST curves, side-channel resistance is a property of the construction
  rather than a discipline requirement on the implementation.
- **Key-generation entropy is platform-dependent.** [`SSFX25519KeyGen()`](#ssfx25519keygen)
  pulls 32 bytes from `SSFPortGetEntropy()` (`ssfport.h`), which reads `/dev/urandom` on
  POSIX and `BCryptGenRandom` on Windows; the raw output *is* the private key, with no
  intermediate PRNG. On embedded targets, port `SSFPortGetEntropy()` to a hardware TRNG, or
  provision keys off-device and use
  [`SSFX25519PubKeyFromPrivKey()`](#ssfx25519pubkeyfromprivkey) at runtime instead.
  **Never** ship this module's `KeyGen` on a target that lacks a real entropy source.
- **Scalar clamping is internal.** Per RFC 7748 §5, before the ladder runs the scalar is
  masked: bits 0–2 of `privKey[0]` are cleared, bit 7 of `privKey[31]` is cleared, and bit
  6 of `privKey[31]` is set. This module performs the clamping inside
  [`_x25519_scalar_mul`](#ssfx25519computesecret) — callers hand in the raw 32 random
  bytes from `SSFX25519KeyGen` (or from any other entropy source) and do not need to do
  anything special. Two consequences worth knowing: (1) the `privKey` bytes the caller
  stores are *not* the clamped scalar — storing the raw bytes and letting the primitive
  re-clamp on each use is correct; (2) private keys that differ only in the three
  low-order bits, or in the two top bits, produce identical public keys and shared
  secrets.
- **Low-order output check.** [`SSFX25519ComputeSecret()`](#ssfx25519computesecret) returns
  `false` if the resulting shared secret is the all-zero value. That happens when the peer
  supplies a point of small order in the twist group (or certain low-order points on the
  main group); multiplying such a point by any clamped scalar yields a predictable result
  rather than a real shared secret. RFC 7748 §6.1 explicitly recommends rejecting this
  outcome to defeat small-subgroup attacks. Other validity concerns (points of
  non-squared-x on the twist) are covered by the contributory-behaviour property of the
  clamped X25519 itself and do not need separate checks.
- **Wire format is little-endian 32 bytes.** Both private and public keys are the raw
  u-coordinate as a little-endian 32-byte integer (RFC 7748 §5), with no prefix byte, no
  length field, no ASN.1 wrapping. Public keys are the u-coordinate of `privKey ·
  basepoint-9`; no y-coordinate is transmitted (X25519 uses only u for both input and
  output, which is half the reason the primitive is fast).
- **The shared secret is not a key.** Pass the 32 bytes through
  [`ssfhkdf`](ssfhkdf.md) (NIST SP 800-56C-style extraction + expansion) to produce
  usable session keys. Do **not** use the raw secret directly as a symmetric key — the
  distribution of valid X25519 outputs is not uniform over `{0,1}²⁵⁶`, and the standard
  warns against using it verbatim.
- **Self-contained.** Does not depend on [`ssfbn`](ssfbn.md) or [`ssfec`](ssfec.md). The
  GF(2²⁵⁵−19) field arithmetic is implemented directly inside this module using 8 × 32-bit
  limbs. Enable X25519 without paying for NIST-curve support if that's all you need.
- **No key-validation entry point.** Unlike [`ssfecdsa`'s
  `SSFECDSAPubKeyIsValid`](ssfecdsa.md#ssfecdsapubkeyisvalid), this module has no separate
  "is this a well-formed key" function. All 32-byte strings are valid X25519 public keys
  on paper — the ones you should reject are the ones that produce an all-zero shared
  secret, which [`SSFX25519ComputeSecret`](#ssfx25519computesecret) reports on its own
  `false` path. Callers do not need to sanity-check the peer's public key beyond passing
  it straight in.

## [↑](#ssfx25519--x25519-key-agreement) Function Reference

<a id="ssfx25519keygen"></a>

### [↑](#functions) [`bool SSFX25519KeyGen()`](#functions)

```c
bool SSFX25519KeyGen(uint8_t privKey[32], uint8_t pubKey[32]);
```

Generate a fresh `(privKey, pubKey)` pair. Reads 32 bytes directly from the platform
entropy source into `privKey`, then computes `pubKey = privKey · basepoint-9` via the
constant-time scalar multiplication. Returns `false` if the entropy source is unavailable.

The raw bytes placed in `privKey` are not clamped — clamping happens internally each time
the primitive uses the scalar. Storing the raw bytes long-term is correct and matches the
way the key is used.

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `privKey` | out | `uint8_t[32]` | Raw 32-byte private key (unclamped). Treat as secret material and zeroize before discard. |
| `pubKey` | out | `uint8_t[32]` | 32-byte public key (little-endian u-coordinate). |

**Returns:** `true` on success; `false` if the platform entropy source is unavailable.

<a id="ex-keygen"></a>

**Example:**

```c
uint8_t priv[SSF_X25519_KEY_SIZE];
uint8_t pub[SSF_X25519_KEY_SIZE];

if (!SSFX25519KeyGen(priv, pub))
{
    /* Platform entropy missing — on a bare-metal target, port
       SSFPortGetEntropy() to a TRNG. */
}
/* pub goes on the wire; priv stays local and secret. */
```

<a id="ssfx25519pubkeyfromprivkey"></a>

### [↑](#functions) [`void SSFX25519PubKeyFromPrivKey()`](#functions)

```c
void SSFX25519PubKeyFromPrivKey(const uint8_t privKey[32], uint8_t pubKey[32]);
```

Derive `pubKey = privKey · basepoint-9` from an existing private key, without touching the
entropy source. Useful when the private key is provisioned from a secure element or
factory-burned at manufacture and the device needs to advertise its public key at boot.

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `privKey` | in | `const uint8_t[32]` | 32-byte private key (raw bytes; clamping is internal). |
| `pubKey` | out | `uint8_t[32]` | 32-byte public key. |

**Returns:** Nothing. Cannot fail for any well-formed 32-byte input — every 32-byte scalar
produces a valid public key after internal clamping.

<a id="ex-derive"></a>

**Example:**

```c
/* Device private key burned into flash at manufacture. */
extern const uint8_t provisioned_priv[32];

uint8_t pub[SSF_X25519_KEY_SIZE];
SSFX25519PubKeyFromPrivKey(provisioned_priv, pub);
/* pub is ready to send to peers, embed in attestations, etc. */
```

<a id="ssfx25519computesecret"></a>

### [↑](#functions) [`bool SSFX25519ComputeSecret()`](#functions)

```c
bool SSFX25519ComputeSecret(const uint8_t privKey[32],
                            const uint8_t peerPubKey[32],
                            uint8_t secret[32]);
```

Compute the shared secret `secret = privKey · peerPubKey` via the constant-time Montgomery
ladder, then reject the result if every output byte is zero (RFC 7748 §6.1 small-subgroup
defence). Returns `true` on success with the shared secret written to `secret`, `false`
otherwise — in which case `secret` should be treated as invalid and discarded (the bytes
may have been written, but are not a usable shared secret).

| Parameter | Direction | Type | Description |
|-----------|-----------|------|-------------|
| `privKey` | in | `const uint8_t[32]` | Local private key. |
| `peerPubKey` | in | `const uint8_t[32]` | Peer's 32-byte public key. |
| `secret` | out | `uint8_t[32]` | Shared secret buffer. |

**Returns:** `true` if the shared secret is non-zero; `false` if the peer supplied a
low-order point that forced the result to zero. On `false`, do not use `secret`.

**The raw output is not a key.** Pass it through [`ssfhkdf`](ssfhkdf.md) to derive actual
keys.

<a id="ex-ecdh"></a>

**Example:**

```c
/* Full X25519 + HKDF handshake: derive two AES-256 session keys from the
   raw Diffie-Hellman output. */
uint8_t shared[SSF_X25519_SECRET_SIZE];

if (!SSFX25519ComputeSecret(myPriv, peerPub, shared))
{
    /* Peer sent a low-order point — abort the handshake. An attacker who
       picks a point of small order learns nothing (shared is zero), but
       continuing would let them bypass key agreement entirely. */
    return;
}

/* shared is 32 bytes of raw ECDH output — not yet usable as a key. */
uint8_t clientKey[32];
uint8_t serverKey[32];

SSFHKDF(SSF_HMAC_HASH_SHA256,
        sessionSalt, sessionSaltLen,
        shared,      sizeof(shared),
        (const uint8_t *)"client write key", 16,
        clientKey,   sizeof(clientKey));
SSFHKDF(SSF_HMAC_HASH_SHA256,
        sessionSalt, sessionSaltLen,
        shared,      sizeof(shared),
        (const uint8_t *)"server write key", 16,
        serverKey,   sizeof(serverKey));

/* Raw secret has served its purpose; wipe it. */
memset(shared, 0, sizeof(shared));
```
